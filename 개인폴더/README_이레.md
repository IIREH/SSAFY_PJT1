## 개인 공부 - NoSql과 mongoDB
### NoSql
Not only Sql의 약자로 
- 일관성을 좀 포기하고(ACID X)
- 데이터를 통으로 저장하는 특징이 있다.

이로 인해
- 속도가 빠르며
- 분산저장에 용이하고
- 데이터를 코드에서 익숙한 형태로 바로 다룰 수 있고
- 스키마가 따로 필요 없어 필드의 이름만 잘 유지하면 됨

### mongoDB
NoSql 중 하나로 Document model을 따름, 그냥 DB가 JSON을 뭉테기로 쌓아놓은 것 같다.
- 용어:
    - Document: Row
    - Collection: Table
    - Collection의 집합: DB 
- Document model: K-V model과 비슷하되 좀 더 발전시켜 V값에 document를 계층적으로 사용 가능(OOP의 composite같은?)

## mongo express 연동
- mongoDB를 웹 환경으로 쉽게 보고 다룰 수 있는 mongo express라는 걸 써보자는 의견에 도입을 시도 
- docker에 mongoDB와 mongo express 컨테이너를 생성해 port를 맞추었으나 express가 mongoDB에 접속을 못함
- 에러 로그를 살피니 connection 자체가 안된다하고 대충 dns, lookup 등 키워드가 보여 도커 네트워크 구성을 찾아봄
- 도커 내부적으로 bridge, host, none 등으로 구분되는데 저 두 컨테이너 모두 기본 bridge network에 연결되는데 왜안되지?
- 공식문서를 보니 기본 bridge는 명시적으로 컨테이너를 지정해주지 않으면 연결이 안된다함
- 그래서 커스텀 네트워크를 만들어 두 컨테이너에 붙였더니 문제해결
- 생각해보니 컨테이너간의 격리를 위해 기본 bridge에서 컨테이너간의 통신 차단이 디폴트인 건 당연하지싶음
- 그리고 간단히 mongoDB 찍먹하며 express로 모니터링해봄

# 1월 17일
## 주말간
docker compose으로 컨테이너들을 묶다가 문제가 있었다. 원인은 의존성이 있는 컨테이너 사이에서 아직 올바르게 작동하지 못하는 컨테이너에 다른 컨테이너가 문을 두들기는 것. 해당 컨테이너의 image에 wait-for-it 같은 스크립트를 넣어주어 해결하려했는데 그냥 기성품으로 나온 이미지를 다시 스크립트 넣어서 재가공하기도 귀찮고, 이게 사용자가 겪는 문제가 아니라 개발자가 겪는 문제고 치명적이지도 않으니.. 그냥 개발자가 도커 올리고 몇 초 기다렸다가 문제되는 컨테이너만 재실행하기로 했다. 

이걸로 터질만한 잠재적 문제점은 해당 컨테이너를 쓰려는데 실은 깜빡하고 그 컨테이너의 재실행을 안해줬다, 정도고 해결책도 거 다시 켜주면 되는 거니까 크게 문제있다고는 생각하지 않아 이렇게 결정했다.

## 공식문서 읽기
개발하면서 항상 옆에 끼고 살아야하는 공식문서..를 읽어가는데 비효율을 만드는 장애물이 있다.

1. 배경지식이 모자라서 말귀를 잘 못알아먹겠고
2. 필요한 항목이 목차 어디에 있는지 찾기도 힘들고
3. 영어

각 케이스마다
1. 기술자들이 당연하게 생각하는 배경지식을 쌓고, *해당 기술이 추구하는 철학과 목표를 잘 이해하여 이 문서를 써내린 사람의 시선에 동기화*할 것에 유의하고
2. 좀 더 많은 걸 겪어서 소위 짬이라는 게 쌓이면 대충 눈에 들어오는 틀이 생길 거 같고 또
3. 영어공부를 해야겠고 (다행히 기술에 관한 글은 영문학은 아니어서 단어나 문장구조가 반복되는 것 같다)

나름 기술문서부터 찾아보고 읽는다 생각했는데 알고보니 초보적인 안내나 기초적인 사용법, 튜토리얼 등의 내용이 아니라면 입구컷 당하는 일이 잦다. 좀만 깊게 들어가면 사실상 어느 블로거가 꼭꼭 씹어 소화해 쓴 글로 기초를 잡고 거기에 레퍼런스라고 달아준 링크로 기술문서를 들어가는 일이 빈번해 참 한심하다.
